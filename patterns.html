Acceptance Patterns

Acceptance is up to the developer. The framework gives the agent governed candidate material; the agent decides whether to accept, reject, review, or refine it.

The Core Principle

Large Language Models act as probabilistic generators, producing outputs that cannot be trusted implicitly. Generative Layers addresses this by quarantining raw outputs as uncommitted candidate material outside the agent's belief base. Using BDI reasoning, the host agent operates as a symbolic gatekeeper—inspecting, peer-reviewing, and auditing candidates before committing them. This ensures that neural suggestions only become active beliefs once they satisfy formal admissibility constraints.

Application-Level Governance Blueprints

The implemented acceptance patterns above are the reusable building blocks. The blueprints below show how a developer can compose those solved patterns and Generative Layers lifecycle commands into larger BDI agent architectures. They are not additional primitive patterns and they are not turnkey applications. Each domain still needs the developer to define the agent roles, domain rules, trusted evidence sources, safety thresholds, retry strategy, and final acceptance logic. The command combinations are kept explicit so the path from architecture to implementation remains visible.

Architectural Paradigms & Diverse Research Directions
Project 1 — Multi-Agent Software Engineering Swarm (Autonomous Plan-Execution Loops)
Domain: Automated Code Generation & Quality Control.
How it works: A decentralized multi-agent BDI system simulating software engineering roles, cooperating through belief-sharing and self-correcting feedback cycles to continuously produce and audit code assets.
Command Combinations:
  • Product Owner Agent: Receives raw specifications and calls call(..., "ANSWER", prompt, "stories") to decompose the requirement into a structured backlog of user stories.
  • Developer Agent: Receives a user story and invokes a model via call(..., "ANSWER", prompt, "code"), immediately quarantining the output inside a candidate(resId) transaction.
  • Tester Agent: Generates corresponding unit tests, compiles the program, and uses check(resId) to verify build integrity and plan execution correctness.
  • Security Auditor Agent: Audits the sandboxed file and registers a policy compliance rating via judge(candId, "auditor", "APPROVE", confidence, rationale).
  • Governance Plan: Evaluates decide(candId) to check admissibility. If inadmissible, it calls reject(candId, "failed audit"), feeding compiler/linter errors back into a new call for a self-correcting retry loop. If admissible, it runs accept(candId, "all checks passed") to commit the code.
Project 2 — Self-Optimizing RAG-based Clinical Assistant (Safe Epistemic Foundations)
Domain: Healthcare & Safety-Critical Diagnostics.
How it works: A diagnostic support system that guards the agent's epistemic belief base, isolating and verifying generated medical recommendations against trusted scientific repositories before committing them as active beliefs.
Command Combinations:
  • Triage Agent: Ingests patient symptoms and calls call(..., "ANSWER", symptoms, "lookup_keys") to generate optimal query terms for clinical databases.
  • Search Agent: Queries the database and asserts the retrieved clinical papers into the agent's local belief base (BDI-level adoption, outside GL).
  • Diagnostic Agent: Dispatches patient telemetry via call(..., "ANSWER", patient_data, "diagnosis,dosage,citations") to draft a diagnostic report, immediately calling candidate(resId) to quarantine it.
  • Safety Review: A strict rule-checking pipeline parses quarantined sub-fields with get(candId, "citations") and get(candId, "dosage"), cross-referencing values against safe medical thresholds and registering a clinical judge(candId, "safety_auditor", verdict, confidence, rationale).
  • Resolution Plan: Deliberates over decide(candId). If inadmissible, it triggers reject(candId, "unsafe dosage") to block state pollution; otherwise, it calls accept(candId, "verified diagnosis") to commit the report.
Project 3 — E-Commerce Arbitrage & Negotiating Agent (Decentralized Game-Theoretic Plan Harmonization)
Domain: Autonomous Trading & Game Theory.
How it works: An automated trading agent that negotiates wholesale supplier contracts, enforcing hard financial spending limits to prevent capital depletion during high-frequency bidding.
Command Combinations:
  • Scouter Agent: Dispatches market trends using call(..., "ANSWER", market_query, "item,price,trend") to identify arbitrage opportunities, asserting validated findings into its beliefs.
  • Negotiator Agent: Negotiates contract bids, calling call(..., "ANSWER", negotiation_prompt, "offer,price") to draft a counter-offer as quarantined candidate material.
  • Safety Guard: Extracts the bid value and invokes judge(candId, "budget_guard", verdict, confidence, rationale) to audit the proposal against the agent's maximum budget limit, then evaluates decide(candId) to verify admissibility.
  • Commit Plan: Verifies the final supplier price with get(candId, "price"). If inadmissible, it triggers reject(candId); if admissible, it runs accept(candId, "within budget") to commit the contract.
Project 4 — Autonomous Incident Response Grid (Self-Healing Cyber-Physical Systems)
Domain: IoT, Energy Logistics, & Emergency Mitigation.
How it works: A self-healing grid-management system modeled using BDI agents, coordinating dynamically to identify cyber-physical anomalies, draft load-routing configurations, and execute plans to bypass physical line faults safely.
Command Combinations:
  • Monitor Agent: Receives real-time telemetry from smart sensors, triggering recovery goals upon registering physical line anomalies.
  • Planner Agent: Formulates alternative load-routing sequences around damaged substations using call(..., "ANSWER", anomaly_data, "route,load_estimate"), quarantining the draft via candidate(resId).
  • Validation Guard: Runs a grid simulation of the proposed path, registering a judge(candId, "load_simulator", verdict, confidence, rationale) to verify that the route remains within safe capacity limits before evaluating decide(candId).
  • Decision Loop: If decide(candId) fails, the plan triggers reject(candId, "overload predicted") and issues a new call to generate a load-shedding route. Once admissible, accept(candId, "safe route confirmed") commits the changes, triggering physical switches.
Project 5 — Interactive NPC Dynamic Dialogue System (Emergent Narrative & Social Interaction)
Domain: Gaming & Creative Entertainment.
How it works: Virtual game characters with distinct personality profiles and social beliefs, generating contextually appropriate dialogue while auditing responses to ensure compliance with established world lore and behavioral bounds.
Command Combinations:
  • Perception Agent: Records player choices and updates social relationship beliefs within the BDI belief base (e.g. player_relation(NPC, player, hostile)).
  • Dialogue Agent: Generates dialogue drafts based on mood and traits via call(..., "ANSWER", context, "dialogue,tone"), quarantining the speech string.
  • Lore Auditor: Evaluates response compliance against established world lore and content guidelines, registering an independent judge(candId, "lore_auditor", verdict, confidence, rationale) before running decide(candId).
  • Adopt & Speak: If admissible, the agent calls accept(candId, "lore compliant") and records the dialogue line into the narrative history before playing the voice line; otherwise, it triggers reject(candId) and runs a conversational retry.
