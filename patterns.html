Acceptance Patterns

Acceptance is up to the developer. The framework gives you the tools — your agent decides what to do with LLM output.

The Core Principle

The LLM is a tool, not a decision-maker. When an agent calls ask, the LLM output becomes candidate material — never adopted beliefs. The agent's own BDI reasoning cycle decides whether to accept or reject each candidate. All examples below use practical reasoning with context-guarded plans — the native BDI decision pattern — rather than imperative if/else branches.

Agent Decides
BDI reasoning, not the LLM
Developer Controls
You write the acceptance logic
Governed Boundary
Nothing enters beliefs unchecked

Pattern 1 — Simple Validation

The simplest pattern: accept if the LLM output matches the expected schema format, reject otherwise. This is the baseline — every other pattern builds on top of it.

When to use: Quick prototyping, low-risk tasks, trusted providers.
What it checks: Schema format only — did the LLM return the right key=value fields?
Limitation: If the LLM returns label=banana when asked about an apple, this pattern still accepts it because the format is valid. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    types governance {
        formula setting(string, string);
        formula configured(boolean);
        formula llm_result(string);
        formula validated(string);
        formula accepted(string);
        formula classified(string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");

    // DOMAIN MODEL
    inference validated(string rid) :- accepted(rid);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !validated(gl.ask("agent1", "classify", "Classify: apple", "convo1"));
    }

    // ACHIEVEMENT: setup (actions only)
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!validated(string rid) : validated(rid) {
        console.println("Already validated: " + rid);
    }

    // ACHIEVEMENT: valid → accept, verify
    rule +!validated(string rid) : gl.valid(rid) == true {
        gl.accept(gl.candidate(rid));
        +accepted(rid);
        +classified(gl.field(rid, "label"));
        ?validated(rid);
        console.println("ACCEPTED: " + gl.field(rid, "label"));
    }

    // ACHIEVEMENT: invalid → reject
    rule +!validated(string rid) {
        console.println("Invalid output → REJECTED");
    }

    // RECOVERY
    rule -!validated(string rid) {
        console.println("Validation FAILED for " + rid);
    }
}

Step-by-step walkthrough
  1. Two competing plans — The agent has two plans for !decide(Rid) with different context guards. The BDI engine selects the plan whose guard matches at runtime.
  2. gl.valid(Rid, true) — Checks if the raw LLM output matched the expected schema (correct key=value fields). This is an automated structural check, not a content judgement.
  3. gl.candidate(Rid, Cid) — Retrieves the candidate ID. Candidates are staged — they exist in a governed buffer, never in the agent's belief base.
  4. gl.accept(Cid) — The agent actively chooses to promote the candidate to adopted knowledge. This is the developer's decision point.
  5. gl.reject(Cid) — Invalid format → reject. The candidate is marked as rejected and traced for audit.

Pattern 2 — Content Inspection

The agent goes beyond schema validation and inspects the actual content of the LLM's answer. Separate context-guarded plans handle known vs. unknown categories.

When to use: When you have business rules about valid answers — the LLM doesn't know your domain constraints.
Key insight: This is where BDI reasoning shines — the agent uses its own beliefs to decide, not the LLM. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    types governance {
        formula setting(string, string);
        formula configured(boolean);
        formula inspected(string);
        formula accepted(string);
        formula known(string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");
    initial known("fruit"), known("vegetable"), known("grain");

    // DOMAIN MODEL
    inference inspected(string rid) :- accepted(rid);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !inspected(gl.invoke_with_beliefs(
            "agent1", "classify", "llm.answer", "ANSWER",
            "Based on the Agent beliefs, classify: apple",
            "label,confidence", "category(apple, fruit)"
        ));
    }

    // ACHIEVEMENT: setup
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!inspected(string rid) : inspected(rid) {
        console.println("Already inspected: " + rid);
    }

    // DECOMPOSITION: valid → check content
    rule +!inspected(string rid) : gl.valid(rid) == true {
        !known_category(gl.candidate(rid), gl.field(rid, "label"));
        +accepted(rid);
        ?inspected(rid);
    }

    // ACHIEVEMENT: invalid
    rule +!inspected(string rid) {
        console.println("Invalid output → REJECTED");
    }

    // ACHIEVEMENT: known → accept
    rule +!known_category(string cid, string label) : known(label) {
        gl.accept(cid);
        console.println("Known category '" + label + "' → ACCEPTED");
    }

    // ACHIEVEMENT: unknown → reject
    rule +!known_category(string cid, string label) : ~known(label) {
        gl.reject(cid);
        console.println("Unknown category '" + label + "' → REJECTED");
    }

    // RECOVERY
    rule -!inspected(string rid) {
        console.println("Inspection FAILED for " + rid);
    }
}

What's different here

The LLM returned valid output — gl.valid passes. But the agent delegates to a sub-goal !check_content with two context-guarded plans. The guard known(Label) checks the agent's own belief base. If the LLM says "legume" but the agent only knows "fruit", "vegetable", and "grain", the second plan fires and rejects. This is practical reasoning — the BDI engine selects the appropriate plan based on the agent's beliefs, exactly as BDI theory intended.

Pattern 3 — Confidence Threshold

The agent asks the LLM to self-assess by including a confidence score (0.0–1.0). Context-guarded plans handle high, medium, and low confidence levels separately.

When to use: Tasks where the LLM can self-assess its certainty (classification, fact retrieval).
Important caveat: LLM self-reported confidence is not always calibrated. Best as one layer in a multi-layer strategy. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;
    module System S;

    types governance {
        formula setting(string, string);
        formula configured(boolean);
        formula threshold(string, string);
        formula assessed(string);
        formula accepted(string);
        formula confidence_tier(string, string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");
    initial threshold("high", "1.0"), threshold("low", "0.0");

    // DOMAIN MODEL
    inference assessed(string rid) :- confidence_tier(rid, string _);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !assessed(gl.invoke(
            "agent1", "classify", "llm.answer", "ANSWER",
            "Classify: apple", "label,confidence"
        ));
    }

    // ACHIEVEMENT: setup
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!assessed(string rid) : assessed(rid) {
        console.println("Already assessed: " + rid);
    }

    // ACHIEVEMENT: high confidence → accept
    rule +!assessed(string rid) : gl.valid(rid) == true & threshold("high", string t) & gl.field(rid, "confidence") == t {
        gl.accept(gl.candidate(rid));
        +accepted(rid);
        +confidence_tier(rid, "high");
        ?assessed(rid);
        console.println("High confidence (" + t + ") → ACCEPTED");
    }

    // ACHIEVEMENT: low confidence → reject, fail
    rule +!assessed(string rid) : gl.valid(rid) == true & threshold("low", string t) & gl.field(rid, "confidence") == t {
        gl.reject(gl.candidate(rid));
        +confidence_tier(rid, "low");
        ?assessed(rid);
        console.println("Low confidence (" + t + ") → REJECTED");
        S.fail();
    }

    // ACHIEVEMENT: medium confidence → escalate (catch-all valid)
    rule +!assessed(string rid) : gl.valid(rid) == true {
        +confidence_tier(rid, "medium");
        ?assessed(rid);
        console.println("Medium confidence → ESCALATING");
    }

    // ACHIEVEMENT: invalid output
    rule +!assessed(string rid) {
        console.println("Invalid output → REJECTED");
    }

    // RECOVERY
    rule -!assessed(string rid) {
        console.println("Assessment FAILED for " + rid);
    }
}

How the three-tier decision works

Three context-guarded plans for !assess_confidence implement a tiered strategy. The BDI engine evaluates guards top-to-bottom: above 0.8 — accept immediately. Between 0.5 and 0.8 — escalate to a verification sub-goal. Below 0.5 — reject outright. The thresholds are up to you — 0.9 for medical, 0.6 for chat. No if/else needed; the BDI engine's plan selection handles it natively.

Confidence: 0.92 → High confidence — ACCEPTED

The LLM returns a confidence score alongside its classification. Scores above 0.8 are accepted, 0.5–0.8 escalated, and below 0.5 rejected. Actual thresholds are configurable by the developer.

Pattern 4 — Cross-LLM Verification

Ask one LLM, then ask a different LLM to verify the answer. Accept only if both providers agree.

When to use: High-stakes decisions, reducing single-provider bias.
How it works: Switch providers with use_provider, send the first answer to the second LLM for verification. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;
    module System S;

    types governance {
        formula setting(string, string);
        formula configured(boolean);
        formula verified(string);
        formula cross_checked(string);
        formula accepted(string);
        formula attempt_count(int);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");
    initial attempt_count(0);

    // DOMAIN MODEL
    inference verified(string rid) :- cross_checked(rid);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !verified(gl.ask("agent1", "classify", "Classify: apple"));
    }

    // ACHIEVEMENT: setup
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!verified(string rid) : verified(rid) {
        console.println("Already verified: " + rid);
    }

    // DECOMPOSITION: valid → cross-check
    rule +!verified(string rid) : gl.valid(rid) == true {
        query(attempt_count(int count));
        -+attempt_count(count + 1);
        !cross_checked(rid);
        ?verified(rid);
    }

    // ACHIEVEMENT: invalid
    rule +!verified(string rid) {
        console.println("Invalid output → ABORTED");
    }

    // DECOMPOSITION: cross-check with second LLM
    rule +!cross_checked(string rid) : gl.valid(rid) == true {
        try {
            !labels_match(rid, gl.field(rid, "label"),
                gl.field(gl.ask("agent1", "verify",
                    "Is this correct? apple = " + gl.field(rid, "label")), "label"));
        } recover {
            console.println("Second LLM call failed");
            S.fail();
        }
    }

    // ACHIEVEMENT: labels match → accept
    rule +!labels_match(string rid, string l1, string l2) : l1 == l2 {
        gl.accept(gl.candidate(rid));
        +accepted(rid);
        +cross_checked(rid);
        console.println("Labels match ('" + l1 + "') → ACCEPTED");
    }

    // ACHIEVEMENT: labels disagree → reject (~= inequality)
    rule +!labels_match(string rid, string l1, string l2) : l1 ~= l2 {
        gl.reject(gl.candidate(rid));
        console.println("Labels DISAGREE ('" + l1 + "' ~= '" + l2 + "') → REJECTED");
    }

    // RECOVERY
    rule -!verified(string rid) {
        console.println("Verification FAILED for " + rid);
    }

    rule -!cross_checked(string rid) {
        console.println("Cross-check FAILED for " + rid);
    }
}

Pattern 5 — Agent-to-Agent Review

One agent generates content, then sends it to a peer agent for review using standard BDI messaging. The peer uses its own beliefs — no LLM involved in the review step.

When to use: Multi-agent systems where agents have different expertise.
How it works: Standard BDI messaging (.send) combined with governed candidates. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;
    module System S;

    types review_types {
        formula setting(string, string);
        formula configured(boolean);
        formula review_timeout(int);
        formula reviewed(string);
        formula peer_approved(string);
        formula review_request(string, string);
        formula review_approved(string);
        formula review_rejected(string, string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");
    initial review_timeout(30000);

    // DOMAIN MODEL
    inference reviewed(string cid) :- peer_approved(cid);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !reviewed(gl.ask("agent1", "classify", "Classify: tomato"));
    }

    // ACHIEVEMENT: setup
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!reviewed(string rid) : reviewed(rid) {
        console.println("[Runner] Already reviewed: " + rid);
    }

    // ACHIEVEMENT: valid → send to reviewer, wait with timeout
    rule +!reviewed(string rid) : gl.valid(rid) == true & review_timeout(int timeout) {
        console.println("[Runner] Sending '" + gl.field(rid, "label") + "' to reviewer...");
        send(request, "reviewer", review_request(gl.field(rid, "label"), gl.candidate(rid)));
        wait(review_approved(gl.candidate(rid)), timeout);
        if (~review_approved(gl.candidate(rid))) {
            console.println("[Runner] Review TIMED OUT → REJECTED");
            gl.reject(gl.candidate(rid));
        }
        ?reviewed(gl.candidate(rid));
    }

    // ACHIEVEMENT: invalid
    rule +!reviewed(string rid) {
        console.println("[Runner] Invalid output → SKIPPED");
    }

    // REACTIVE: peer approved
    rule @message(inform, string sender, review_approved(string cid)) {
        +peer_approved(cid);
        gl.accept(cid);
        console.println("[Runner] Peer approved → ACCEPTED");
    }

    // REACTIVE: peer rejected
    rule @message(inform, string sender, review_rejected(string cid, string reason)) {
        gl.reject(cid);
        console.println("[Runner] Peer rejected: " + reason + " → REJECTED");
    }

    // RECOVERY
    rule -!reviewed(string rid) {
        console.println("[Runner] Review FAILED for " + rid);
    }
}

Pattern 6 — Belief Consistency Check

The agent compares the LLM's answer against its existing beliefs using context guards. Contradictions are rejected, consistent answers confirmed, and new knowledge is adopted.

When to use: When the agent has prior knowledge and must not adopt contradictions.
Key idea: BDI agents have beliefs — use them as a validation layer.

    module Console console;

        formula setting(string, string);
        formula configured(boolean);
        formula consistent(string);
        formula belief_checked(string);
        formula category(string, string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");

    rule +!consistent(string rid) : consistent(rid) {
        console.println("Already checked: " + rid);
    }

    rule +!consistent(string rid) {

    rule +!belief_checked(string item, string label, string cid) : category(item, label) {

    rule +!belief_checked(string item, string label, string cid) : category(item, string existing) & ~category(item, label) {

    }

    // ACHIEVEMENT: no prior belief → adopt new knowledge
    rule +!belief_checked(string item, string label, string cid) : ~category(item, string any) {
        gl.accept(cid);
        +category(item, label);
        console.println("No prior belief for " + item + " → NEW KNOWLEDGE: " + label);
    }

    // RECOVERY
    rule -!consistent(string rid) {
        console.println("Consistency check FAILED for " + rid);
    }

How majority voting reduces bias

Four context-guarded plans for !vote cover all consensus cases. The BDI engine checks pairwise equality: if any two labels match, that pair's answer wins. If all three disagree, the fourth guard fires and nobody's answer is accepted. Each LLM call goes through the full governance pipeline independently. This is the most defensive pattern — useful for critical decisions.

Pattern 8 — Iterative Refinement

The agent generates an answer, critiques it, then refines it. Each round goes through governance. Accept only the final refined version.

When to use: Complex tasks like writing, analysis, or planning where first drafts are rarely good enough.
How it works: Generate → Critique → Refine → Accept. Each step is governed. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    types governance {
        formula setting(string, string);
        formula configured(boolean);
        formula refined(string);
        formula critiqued(string);
        formula improved(string);
        formula accepted_final(boolean);
        formula pipeline_step(string);
    }

    initial setting("model", "gpt-oss-120b"), setting("provider", "cerebras");

    // DOMAIN MODEL
    inference refined(string rid) :- accepted_final(true);

    // DECOMPOSITION: main only adopts subgoals
    rule +!main(list args) {
        !configured(true);
        !refined(gl.ask("agent1", "draft", "Write a 3-sentence summary of photosynthesis"));
    }

    // ACHIEVEMENT: setup
    rule +!configured(boolean x) : setting("model", string m) & setting("provider", string p) {
        gl.configure("model", m);
        gl.use_provider(p);
    }

    // SERENDIPITY
    rule +!refined(string rid) : refined(rid) {
        console.println("Already refined: " + rid);
    }

    // DECOMPOSITION: refine = critique + improve + accept
    rule +!refined(string rid) : gl.valid(rid) == true {
        !critiqued(rid);
        !improved(rid);
        !accepted_final(rid);
        ?refined(rid);
    }

    // ACHIEVEMENT: invalid draft
    rule +!refined(string rid) {
        -+pipeline_step("draft_failed");
        console.println("Draft invalid → ABORTED");
    }

    // ACHIEVEMENT: critique
    rule +!critiqued(string draftRid) : gl.valid(draftRid) == true {
        -+pipeline_step("critiquing");
        +critiqued(gl.ask("agent1", "critique", "Critique the summary of photosynthesis"));
        console.println("Draft → critique requested");
    }

    // SERENDIPITY
    rule +!critiqued(string rid) : critiqued(rid) {
        console.println("Already critiqued");
    }

    // ACHIEVEMENT: improve
    rule +!improved(string rid) : critiqued(string critiqueRid) & gl.valid(critiqueRid) == true {
        -+pipeline_step("refining");
        +improved(gl.ask("agent1", "refine", "Improve the summary based on feedback"));
        console.println("Critique → refinement requested");
    }

    // SERENDIPITY
    rule +!improved(string rid) : improved(rid) {
        console.println("Already improved");
    }

    // ACHIEVEMENT: accept final
    rule +!accepted_final(string rid) : improved(string finalRid) & gl.valid(finalRid) == true {
        gl.accept(gl.candidate(finalRid));
        +accepted_final(true);
        -+pipeline_step("accepted");
        console.println("Draft → Critique → Refine → ACCEPTED");
    }

    // RECOVERY at each level
    rule -!refined(string rid) {
        -+pipeline_step("failed");
        console.println("Refinement pipeline FAILED for " + rid);
    }

    rule -!critiqued(string rid) {
        -+pipeline_step("critique_failed");
        console.println("Critique FAILED → pipeline aborted");
    }

    rule -!improved(string rid) {
        -+pipeline_step("refine_failed");
        console.println("Refinement FAILED → pipeline aborted");
    }
}
