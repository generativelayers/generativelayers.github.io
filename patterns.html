Acceptance Patterns

Acceptance is up to the developer. The framework gives you the tools — your agent decides what to do with LLM output.

The Core Principle

The LLM is a tool, not a decision-maker. When an agent calls gl.ask, the LLM output becomes candidate material — never adopted beliefs. The agent's own BDI reasoning cycle decides whether to accept or reject each candidate.

Agent Decides
BDI reasoning, not the LLM
Developer Controls
You write the acceptance logic
Governed Boundary
Nothing enters beliefs unchecked

Pattern 1 — Simple Validation

The simplest pattern: accept if the LLM output matches the expected schema format, reject otherwise. This is the baseline — every other pattern builds on top of it.

When to use: Quick prototyping, low-risk tasks, trusted providers.
What it checks: Schema format only — did the LLM return the right key=value fields?
What it does NOT check: Whether the answer is actually correct or meaningful.
Limitation: If the LLM returns label=banana when asked about an apple, this pattern still accepts it because the format is valid. 
// Jason — accept if format is valid, reject otherwise
+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "label", Label);
    gl.candidate(Rid, Cid);
    gl.accept(Cid);
    .print("Accepted: ", Label).

+!decide(Rid) : gl.valid(Rid, false) <-
    gl.candidate(Rid, Cid);
    gl.reject(Cid);
    .print("Rejected: invalid format").
Step-by-step walkthrough
  1. gl.valid(Rid, true) — The framework already called the LLM and validated the raw output against the expected schema. This guard checks: did the output contain all required fields (e.g. label=...) in the correct format? This is an automated structural check, not a content judgement.
  2. gl.field(Rid, "label", Label) — Extracts the value of the label field from the validated response. At this point the agent has the LLM's answer as a variable but has not yet adopted it.
  3. gl.candidate(Rid, Cid) — Retrieves the candidate ID associated with this result. Candidates are staged — they exist in a governed buffer, never in the agent's belief base.
  4. gl.accept(Cid) — The agent actively chooses to promote this candidate to adopted knowledge. This is the developer's decision point — you could add any logic before this line.
  5. gl.reject(Cid) — If validation failed, the agent rejects. The candidate is marked as rejected and never enters the belief base. The trace records what happened and why.

Pattern 2 — Content Inspection

The agent goes beyond schema validation and inspects the actual content of the LLM's answer. The key difference from Pattern 1: the agent applies domain-specific business rules before deciding. The LLM's output might be perfectly formatted but still wrong for the agent's context.

When to use: When you have business rules about valid answers — the LLM doesn't know your domain constraints.
Examples: Answer must be in a known category set, must not contain sensitive keywords, must match a regex pattern, must be within a numeric range.
Key insight: This is where BDI reasoning shines — the agent uses its own beliefs and rules, not the LLM, to decide. 
// Jason — accept only if the label is a known category
+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "label", Label);
    gl.candidate(Rid, Cid);

    // Check against agent's own knowledge
    if (Label == "fruit" | Label == "vegetable" | Label == "grain") {
        gl.accept(Cid);
        .print("Accepted known category: ", Label);
    } else {
        gl.reject(Cid);
        .print("Rejected unknown category: ", Label);
    }.
// Jason — reject if the response contains forbidden content
+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "response", Response);
    gl.candidate(Rid, Cid);

    if (.substring("password", Response)) {
        gl.reject(Cid);
        .print("Rejected: contains sensitive content");
    } else {
        gl.accept(Cid);
    }.
What's different here

The LLM returned valid output in both examples — gl.valid passes. But the agent's own logic adds a second gate. In the first example, the agent checks if the label is one of three known categories ("fruit", "vegetable", "grain"). If the LLM invents a new category like "legume", the agent rejects it — even though the LLM was technically correct. In the second example, the agent scans for sensitive keywords. This is the developer deciding what's acceptable, not the LLM. You can combine multiple checks: keyword filtering + category validation + length constraints — all before calling gl.accept.

Pattern 3 — Confidence Threshold

Instead of blindly accepting any valid answer, the agent asks the LLM to self-assess by including a confidence score (0.0–1.0) in its response. The agent then uses this score as a decision threshold — high confidence is accepted, medium triggers further verification, and low is rejected outright.

When to use: Tasks where the LLM can self-assess its certainty (classification, fact retrieval, translation).
Schema requires: Your prompt must ask for a confidence field (0.0–1.0) alongside the answer. The schema validation ensures this field exists.
Important caveat: LLM self-reported confidence is not always calibrated. This pattern works best as one layer in a multi-layer strategy, not as the sole gate. 
// Jason — accept only if confidence > 0.8
+!classify <-
    gl.configure("model", "gemini-2.0-flash");
    gl.ask("agent1", "classify",
        "Classify this item. Return label and confidence (0-1).\nItem: apple", Rid).

+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "label", Label);
    gl.field(Rid, "confidence", Conf);
    gl.candidate(Rid, Cid);

    if (Conf > 0.8) {
        gl.accept(Cid);
        .print("High confidence — accepted: ", Label, " (", Conf, ")");
    } else if (Conf > 0.5) {
        .print("Medium confidence — requesting verification: ", Label);
        !verify(Label, Cid);
    } else {
        gl.reject(Cid);
        .print("Low confidence — rejected: ", Label, " (", Conf, ")");
    }.
How the three-tier decision works

The prompt explicitly asks the LLM to return a confidence field. The schema ensures this field is present. Then the agent's BDI logic applies a three-tier strategy: above 0.8 — the agent trusts the answer and accepts immediately. Between 0.5 and 0.8 — the agent is uncertain, so it triggers a sub-goal (!verify) which could ask a second LLM, consult a peer agent, or check a database. Below 0.5 — the agent considers this too risky and rejects outright. The thresholds are entirely up to you. You could use 0.9 for medical applications or 0.6 for casual chat. The framework doesn't impose any threshold — the agent's reasoning does.

Pattern 4 — Cross-LLM Verification

Ask one LLM, then ask a different LLM to verify the answer. Accept only if both providers agree.

When to use: High-stakes decisions, reducing single-provider bias.
How it works: Switch providers with gl.use_provider, send the first answer to the second LLM for verification. 
// Jason — ask Gemini, then verify with OpenAI

// Step 1: Ask Gemini to classify
+!classify <-
    gl.use_provider("gemini");
    gl.configure("model", "gemini-2.0-flash");
    gl.ask("agent1", "classify", "Classify: apple", GeminiRid).

// Step 2: Send Gemini's answer to OpenAI for verification
+!decide(GeminiRid) : gl.valid(GeminiRid, true) <-
    gl.field(GeminiRid, "label", GeminiLabel);
    +gemini_answer(GeminiRid, GeminiLabel);        // remember it

    gl.use_provider("openai");
    gl.configure("model", "gpt-4o");
    gl.ask("agent1", "verify",
        .concat("Is this classification correct? Item: apple → ", GeminiLabel,
                ". Reply with: agreement=yes or agreement=no, and label=your_answer"),
        OpenAIRid).

// Step 3: Compare both answers
+!decide(OpenAIRid) : gl.valid(OpenAIRid, true)
    & gemini_answer(OrigRid, GeminiLabel) <-
    gl.field(OpenAIRid, "agreement", Agreement);
    gl.candidate(OrigRid, OrigCid);

    if (Agreement == "yes") {
        gl.accept(OrigCid);
        .print("Both LLMs agree: ", GeminiLabel, " ✓");
    } else {
        gl.reject(OrigCid);
        gl.field(OpenAIRid, "label", AltLabel);
        .print("LLMs disagree! Gemini: ", GeminiLabel, " vs OpenAI: ", AltLabel);
    }.
How cross-verification works

The agent uses gl.use_provider to switch between different LLM backends at runtime. Step 1: Ask Gemini to classify the item. The agent stores the answer as a belief (+gemini_answer) but does not accept it yet. Step 2: Switch to OpenAI and ask it to verify Gemini's answer. The agent constructs a verification prompt that includes Gemini's label. Step 3: When OpenAI responds, the agent checks the agreement field. If both providers agree, the original Gemini candidate is accepted. If they disagree, it's rejected. The key insight: the agent orchestrates both LLMs — neither LLM knows about the other. The agent is the decision-maker, using independent opinions as evidence.

Pattern 5 — Agent-to-Agent Review

One agent generates content, then sends it to a peer agent for review. The peer uses its own beliefs and reasoning to approve or reject — no LLM involved in the review step.

When to use: Multi-agent systems where agents have different expertise or knowledge.
How it works: Standard BDI messaging (.send) combined with governed candidates. 
// ─── Agent A: generates and requests peer review ───

+!classify <-
    gl.ask("agentA", "classify", "Classify: tomato", Rid).

+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "label", Label);
    // Don't accept yet — ask Agent B to verify
    .send(agentB, achieve, review(Label, Rid)).

// Agent B approved
+approved(Rid) <-
    gl.candidate(Rid, Cid);
    gl.accept(Cid);
    .print("Peer-approved and accepted").

// Agent B rejected
+disapproved(Rid, Reason) <-
    gl.candidate(Rid, Cid);
    gl.reject(Cid);
    .print("Peer rejected: ", Reason).


// ─── Agent B: reviews using its own beliefs ───

+!review(Label, Rid) <-
    // Agent B checks against its own knowledge base
    if (known_food(Label)) {
        .send(agentA, tell, approved(Rid));
    } else {
        .send(agentA, tell, disapproved(Rid, "not in my knowledge base"));
    }.
How peer review works

This pattern separates the roles of generator and reviewer across two autonomous agents. Agent A calls the LLM and gets a candidate, but instead of deciding alone, it sends the extracted label to Agent B via .send(agentB, achieve, review(...)) — standard BDI inter-agent messaging. Agent B has its own knowledge base (e.g. known_food("fruit")) and applies its own reasoning — no LLM involved in the review. Agent B sends back an approved or disapproved message. Agent A reacts to the verdict and only then calls gl.accept or gl.reject. This means neither agent blindly trusts the LLM or each other — each applies its own governance.

Pattern 6 — Belief Consistency Check

The agent compares the LLM's answer against its existing beliefs. Contradictions are rejected, consistent answers are accepted.

When to use: When the agent has prior knowledge and must not adopt contradictions.
Key idea: BDI agents have beliefs — use them as a validation layer. 
// Jason — agent has existing beliefs about food categories
category("apple", "fruit").
category("carrot", "vegetable").
category("rice", "grain").

+!decide(Rid) : gl.valid(Rid, true) <-
    gl.field(Rid, "item", Item);
    gl.field(Rid, "label", Label);
    gl.candidate(Rid, Cid);

    // Check if I already have a belief about this item
    if (category(Item, ExistingLabel)) {
        if (Label == ExistingLabel) {
            gl.accept(Cid);
            .print("Confirmed my belief: ", Item, " = ", Label);
        } else {
            gl.reject(Cid);
            .print("Contradicts my belief! I know ", Item, " = ",
                   ExistingLabel, ", LLM said: ", Label);
        };
    } else {
        // No prior belief — accept as new knowledge
        gl.accept(Cid);
        .print("New knowledge adopted: ", Item, " = ", Label);
    }.
Why belief consistency matters

BDI agents reason based on their beliefs. If an LLM tells the agent "apple = vegetable" but the agent already believes "apple = fruit", adopting the new belief would create a contradiction — breaking the agent's rational decision-making. This pattern prevents that. The agent checks its belief base before accepting. If the LLM agrees with existing beliefs, the answer is confirmed. If it contradicts, it's rejected. And if the agent has no prior belief, the LLM output is treated as new knowledge and adopted. This is the most BDI-native pattern — it uses the agent's existing knowledge as a validation layer, exactly as BDI theory suggests.

Pattern 7 — Majority Voting

Ask multiple LLMs the same question. Accept only the answer that a majority agrees on.

When to use: Critical decisions where no single LLM should be trusted alone.
How it works: Collect answers from N providers, tally votes, accept the majority winner. 
// Jason — ask 3 providers, accept the majority answer

+!classify <-
    // Ask all three providers
    gl.use_provider("gemini");
    gl.ask("agent1", "classify", "Classify: tomato", R1);
    gl.use_provider("openai");
    gl.ask("agent1", "classify", "Classify: tomato", R2);
    gl.use_provider("anthropic");
    gl.ask("agent1", "classify", "Classify: tomato", R3);

    // Collect answers
    gl.field(R1, "label", L1);
    gl.field(R2, "label", L2);
    gl.field(R3, "label", L3);

    // Majority vote
    if (L1 == L2 | L1 == L3) {
        +winner(L1, R1);
    } else if (L2 == L3) {
        +winner(L2, R2);
    } else {
        .print("No consensus — all three disagree!");
    }.

+winner(Label, Rid) <-
    gl.candidate(Rid, Cid);
    gl.accept(Cid);
    .print("Majority agrees: ", Label).
How majority voting reduces bias

Each LLM has different training data and biases. By asking the same question to three independent providers (Gemini, OpenAI, Anthropic), the agent collects three independent opinions. If at least two agree, the agent accepts that answer. If all three disagree, nobody's answer is accepted — the agent can escalate, retry, or ask a human. The agent uses gl.use_provider to switch providers between calls. Each call goes through the full governance pipeline independently (validation, candidate creation). The agent only accepts the winning candidate from the majority. This is the most defensive pattern — useful for critical decisions like medical triage, financial classification, or safety assessments.

Pattern 8 — Iterative Refinement

The agent generates an answer, critiques it, then refines it. Each round goes through governance. Accept only the final refined version.

When to use: Complex tasks like writing, analysis, or planning where first drafts are rarely good enough.
How it works: Generate → Critique → Refine → Accept. Each step is governed. 
// Jason — generate, critique, refine, then accept

// Step 1: Generate a draft
+!write_report <-
    gl.ask("agent1", "draft",
        "Write a 3-sentence summary of photosynthesis", DraftRid).

// Step 2: Critique the draft
+!decide(DraftRid) : gl.valid(DraftRid, true) <-
    gl.field(DraftRid, "text", Draft);
    gl.ask("agent1", "critique",
        .concat("Critique this summary. List problems:\n", Draft), CritiqueRid).

// Step 3: Refine using the critique
+!decide(CritiqueRid) : gl.valid(CritiqueRid, true) <-
    gl.field(CritiqueRid, "feedback", Feedback);
    gl.field(DraftRid, "text", Draft);
    gl.ask("agent1", "refine",
        .concat("Improve this:\n", Draft, "\nBased on:\n", Feedback), FinalRid).

// Step 4: Accept only the refined version
+!decide(FinalRid) : gl.valid(FinalRid, true) <-
    gl.candidate(FinalRid, Cid);
    gl.accept(Cid);
    gl.field(FinalRid, "text", FinalText);
    .print("Final accepted version:\n", FinalText).
How iterative refinement improves quality

The agent orchestrates a three-step pipeline where each step feeds into the next through the governance boundary. Step 1 (Draft): The LLM generates a first attempt. The agent extracts the text but does NOT accept it. Step 2 (Critique): The agent feeds the draft to the LLM and asks for criticism. Again, the critique is governed — validated and extracted as candidate material. Step 3 (Refine): The agent combines the original draft with the critique and asks for an improved version. Only this final, refined output is accepted as knowledge. Each step goes through the full governance pipeline: schema validation, candidate creation, and traceability. The draft and critique are intermediate artifacts — never adopted. This pattern is especially powerful because the agent controls the pipeline, deciding when quality is good enough to stop iterating.
