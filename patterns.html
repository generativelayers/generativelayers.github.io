Acceptance Patterns

Acceptance is up to the developer. The framework gives you the tools — your agent decides what to do with LLM output.

The Core Principle

The LLM is a tool, not a decision-maker. When an agent calls ask, the LLM output becomes candidate material — never adopted beliefs. The agent's own BDI reasoning cycle decides whether to accept or reject each candidate. All examples below use practical reasoning with context-guarded plans — the native BDI decision pattern — rather than imperative if/else branches.

Agent Decides
BDI reasoning, not the LLM
Developer Controls
You write the acceptance logic
Governed Boundary
Nothing enters beliefs unchecked

Pattern 1 — Simple Validation

The simplest pattern: accept if the LLM output matches the expected schema format, reject otherwise. This is the baseline — every other pattern builds on top of it.

When to use: Quick prototyping, low-risk tasks, trusted providers.
What it checks: Schema format only — did the LLM return the right key=value fields?
Limitation: If the LLM returns label=banana when asked about an apple, this pattern still accepts it because the format is valid. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    rule +!main(list args) {
        console.println("=== Pattern 1: Simple Validation ===");
        gl.use_provider();
        string rid = gl.ask("agent1", "classify", "Classify: apple");
        !decide(rid);
    }

    rule +!decide(string rid) : gl.valid(rid) == true {
        string cid = gl.candidate(rid);
        gl.accept(cid);
        console.println("Valid output → ACCEPTED: " + gl.field(rid, "label"));
    }

    rule +!decide(string rid) : gl.valid(rid) == false {
        console.println("Invalid output → REJECTED");
    }
}

Step-by-step walkthrough
  1. Two competing plans — The agent has two plans for !decide(Rid) with different context guards. The BDI engine selects the plan whose guard matches at runtime.
  2. gl.valid(Rid, true) — Checks if the raw LLM output matched the expected schema (correct key=value fields). This is an automated structural check, not a content judgement.
  3. gl.candidate(Rid, Cid) — Retrieves the candidate ID. Candidates are staged — they exist in a governed buffer, never in the agent's belief base.
  4. gl.accept(Cid) — The agent actively chooses to promote the candidate to adopted knowledge. This is the developer's decision point.
  5. gl.reject(Cid) — Invalid format → reject. The candidate is marked as rejected and traced for audit.

Pattern 2 — Content Inspection

The agent goes beyond schema validation and inspects the actual content of the LLM's answer. Separate context-guarded plans handle known vs. unknown categories.

When to use: When you have business rules about valid answers — the LLM doesn't know your domain constraints.
Key insight: This is where BDI reasoning shines — the agent uses its own beliefs to decide, not the LLM. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    types pattern_types {
        formula known(string);
    }

    initial known("fruit"), known("vegetable"), known("grain");

    rule +!main(list args) {
        console.println("=== Pattern 2: Content Inspection ===");
        gl.use_provider();
        string rid = gl.ask("agent1", "classify", "Classify: apple");
        !decide(rid);
    }

    rule +!decide(string rid) : gl.valid(rid) == true {
        string label = gl.field(rid, "label");
        string cid = gl.candidate(rid);
        !check_content(cid, label);
    }

    rule +!check_content(string cid, string label) : known(label) {
        gl.accept(cid);
        console.println("Known category '" + label + "' → ACCEPTED");
    }

    rule +!check_content(string cid, string label) : ~known(label) {
        gl.reject(cid);
        console.println("Unknown category '" + label + "' → REJECTED");
    }
}

Pattern 3 — Confidence Threshold

The agent asks the LLM to self-assess by including a confidence score (0.0–1.0). Context-guarded plans handle high, medium, and low confidence levels separately.

When to use: Tasks where the LLM can self-assess its certainty (classification, fact retrieval).
Important caveat: LLM self-reported confidence is not always calibrated. Best as one layer in a multi-layer strategy. 
agent PatternRunner {
    module gl.adapter.astra.AstraAdapter gl;
    module Console console;

    rule +!main(list args) {
        console.println("=== Pattern 3: Confidence Threshold ===");
        gl.use_provider();
        string rid = gl.ask("agent1", "classify", "Classify: apple");
        !decide(rid);
    }

    rule +!decide(string rid) : gl.valid(rid) == true {
        string cid = gl.candidate(rid);
        string conf = gl.field(rid, "confidence");
        !assess(cid, conf);
    }

    rule +!assess(string cid, string conf) : conf == "1.0" {
        gl.accept(cid);
        console.println("High confidence (1.0) → ACCEPTED");
    }

    rule +!assess(string cid, string conf) : conf == "0.0" {
        gl.reject(cid);
        console.println("Low confidence (0.0) → REJECTED");
    }

    rule +!assess(string cid, string conf) {
        console.println("Medium confidence (" + conf + ") → ESCALATING");
    }
}

How the three-tier decision works

Three context-guarded plans for !assess_confidence implement a tiered strategy. The BDI engine evaluates guards top-to-bottom: above 0.8 — accept immediately. Between 0.5 and 0.8 — escalate to a verification sub-goal. Below 0.5 — reject outright. The thresholds are up to you — 0.9 for medical, 0.6 for chat. No if/else needed; the BDI engine's plan selection handles it natively.

Pattern 4 — Cross-LLM Verification

Ask one LLM, then ask a different LLM to verify the answer. Accept only if both providers agree.

When to use: High-stakes decisions, reducing single-provider bias.
How it works: Switch providers with use_provider, send the first answer to the second LLM for verification. 
// Step 1: Ask Gemini
rule +!classify() {
    gl.use_provider("gemini");
    string rid = gl.ask("agent1", "classify", "Classify: apple");
    !verify_with_second(rid);
}

// Step 2: Send to OpenAI for verification
rule +!verify_with_second(string rid) : gl.valid(rid) == true {
    string label = gl.field(rid, "label");
    gl.use_provider("openai");
    string vrid = gl.ask("agent1", "verify",
        "Is this correct? apple = " + label);
    !compare_results(rid, vrid);
}

// Step 3: Both agree → accept
rule +!compare_results(string orig, string vrid)
    : gl.field(vrid, "agreement") == "yes" {
    string cid = gl.candidate(orig);
    gl.accept(cid);
}

// Step 3: Disagree → reject
rule +!compare_results(string orig, string vrid)
    : gl.field(vrid, "agreement") == "no" {
    string cid = gl.candidate(orig);
    gl.reject(cid);
}

The agent orchestrates two independent LLMs. Step 1: Ask Gemini. Step 2: Switch to OpenAI and pass Gemini's answer for verification. Step 3: Two context-guarded plans for !compare_results handle agreement vs. disagreement. Neither LLM knows about the other — the agent is the decision-maker using independent opinions as evidence.

Pattern 5 — Agent-to-Agent Review

One agent generates content, then sends it to a peer agent for review using standard BDI messaging. The peer uses its own beliefs — no LLM involved in the review step.

When to use: Multi-agent systems where agents have different expertise.
How it works: Standard BDI messaging (.send) combined with governed candidates. 
// ─── Agent A: generates and requests peer review ───
+!classify <-
    gl.ask("agentA", "classify", "Classify: tomato", Rid);
    !request_review(Rid).

+!request_review(Rid)
    :   gl.valid(Rid, true)
    <-  gl.field(Rid, "label", Label);
        .send(agentB, achieve, review(Label, Rid)).

// Context guard: peer approved → accept
+approved(Rid) <-
    gl.candidate(Rid, Cid);
    gl.accept(Cid).

// Context guard: peer rejected → reject
+disapproved(Rid, Reason) <-
    gl.candidate(Rid, Cid);
    gl.reject(Cid).

// ─── Agent B: reviews using its own beliefs ───
known_food("fruit"). known_food("vegetable").

+!review(Label, Rid)
    :   known_food(Label)
    <-  .send(agentA, tell, approved(Rid)).

+!review(Label, Rid)
    :   not known_food(Label)
    <-  .send(agentA, tell, disapproved(Rid, "not in my knowledge base")).
// ─── Agent A (JaCaMo): uses CArtAgO artifact ───
+gl_result(Rid, Outcome, true) <-
    field(Rid, "label", Label);
    .send(agentB, achieve, review(Label, Rid)).

+approved(Rid) <-
    candidate(Rid, Cid);
    accept(Cid).

+disapproved(Rid, Reason) <-
    candidate(Rid, Cid);
    reject(Cid).

// ─── Agent B: same review plans as Jason ───
known_food("fruit"). known_food("vegetable").

+!review(Label, Rid)
    :   known_food(Label)
    <-  .send(agentA, tell, approved(Rid)).

+!review(Label, Rid)
    :   not known_food(Label)
    <-  .send(agentA, tell, disapproved(Rid, "not in my knowledge base")).
How peer review works

Agent A generates a candidate but does not decide alone. It sends the label to Agent B via standard BDI messaging. Agent B has two context-guarded plans for !review — one fires if the label matches a known food belief, the other if it doesn't. Agent A reacts to the verdict via belief triggers (+approved / +disapproved). Neither agent blindly trusts the LLM or each other — each applies its own governance. Note: ASTRA does not natively support inter-agent messaging, so this pattern is shown for Jason and JaCaMo only.

Pattern 6 — Belief Consistency Check

The agent compares the LLM's answer against its existing beliefs using context guards. Contradictions are rejected, consistent answers confirmed, and new knowledge is adopted.

When to use: When the agent has prior knowledge and must not adopt contradictions.
Key idea: BDI agents have beliefs — use them as a validation layer. 
// Existing beliefs
+category("apple", "fruit"). +category("carrot", "vegetable").

rule +!decide(string rid) : gl.valid(rid) == true {
    string item = gl.field(rid, "item");
    string label = gl.field(rid, "label");
    string cid = gl.candidate(rid);
    !check_belief(cid, item, label);
}

// Matches existing belief → confirmed
rule +!check_belief(string cid, string item, string label)
    : category(item, label) {
    gl.accept(cid);
    console.println("Confirmed: " + item + " = " + label);
}

// Contradicts existing belief → rejected
rule +!check_belief(string cid, string item, string label)
    : category(item, string existing) & existing != label {
    gl.reject(cid);
    console.println("Contradicts belief!");
}

// No prior belief → new knowledge
rule +!check_belief(string cid, string item, string label)
    : ~category(item, string any) {
    gl.accept(cid);
    console.println("New knowledge adopted: " + item);
}
// Existing beliefs
category("apple", "fruit"). category("carrot", "vegetable").

+!decide(Rid)
    :   gl.valid(Rid, true)
    <-  gl.field(Rid, "item", Item);
        gl.field(Rid, "label", Label);
        gl.candidate(Rid, Cid);
        !check_belief(Cid, Item, Label).

// Matches existing belief → confirmed
+!check_belief(Cid, Item, Label)
    :   category(Item, Label)
    <-  gl.accept(Cid);
        .print("Confirmed: ", Item, " = ", Label).

// Contradicts existing belief → rejected
+!check_belief(Cid, Item, Label)
    :   category(Item, Existing) & Label \== Existing
    <-  gl.reject(Cid);
        .print("Contradicts belief! I know ", Item, "=", Existing).

// No prior belief → new knowledge
+!check_belief(Cid, Item, Label)
    :   not category(Item, _)
    <-  gl.accept(Cid);
        .print("New knowledge adopted: ", Item, "=", Label).
// Existing beliefs
category("apple", "fruit"). category("carrot", "vegetable").

+gl_result(Rid, Outcome, true) <-
    field(Rid, "item", Item);
    field(Rid, "label", Label);
    candidate(Rid, Cid);
    !check_belief(Cid, Item, Label).

    :   category(Item, Label)
    <-  accept(Cid);
        .print("Confirmed: ", Item, " = ", Label).

+!check_belief(Cid, Item, Label)
    :   category(Item, Existing) & Label \== Existing

+!check_belief(Cid, Item, Label)

Why belief consistency matters

Three context-guarded plans for !check_belief handle three scenarios. The BDI engine checks the agent's belief base: category(Item, Label) matches → confirmed. category(Item, Existing) exists but differs → contradiction rejected. No category belief → adopted as new knowledge. This is the most BDI-native pattern — it uses the agent's existing knowledge exactly as BDI theory intended.

Pattern 7 — Majority Voting

Ask multiple LLMs the same question. Accept only the answer that a majority agrees on.

When to use: Critical decisions where no single LLM should be trusted alone.
How it works: Collect answers from N providers, tally votes with context-guarded plans.

How majority voting reduces bias

Four context-guarded plans for !vote cover all consensus cases. The BDI engine checks pairwise equality: if any two labels match, that pair's answer wins. If all three disagree, the fourth guard fires and nobody's answer is accepted. Each LLM call goes through the full governance pipeline independently. This is the most defensive pattern — useful for critical decisions.

Expected Result 
=== Pattern 7: Majority Voting ===

Majority agrees on 'vegetable' → ACCEPTED

Three LLM calls are made (can use different providers). Labels are compared pairwise. If any two agree, the majority answer is accepted. If all three disagree, no decision is made. This is the most defensive pattern.

Pattern 8 — Iterative Refinement

The agent generates an answer, critiques it, then refines it. Each round goes through governance. Accept only the final refined version.

When to use: Complex tasks like writing, analysis, or planning where first drafts are rarely good enough.
How it works: Generate → Critique → Refine → Accept. Each step is governed. 
// Step 1: Generate a draft
+!write_report <-
    gl.ask("agent1", "draft",
        "Write a 3-sentence summary of photosynthesis", DraftRid);
    !critique_draft(DraftRid).

// Step 2: Critique the draft (valid draft only)
+!critique_draft(DraftRid)
    :   gl.valid(DraftRid, true)
    <-  gl.field(DraftRid, "text", Draft);
        gl.ask("agent1", "critique",
            .concat("Critique this:\n", Draft), CritiqueRid);
        !refine_draft(DraftRid, CritiqueRid).

// Step 3: Refine using the critique (valid critique only)
+!refine_draft(DraftRid, CritiqueRid)
    :   gl.valid(CritiqueRid, true)
    <-  gl.field(DraftRid, "text", Draft);
        gl.field(CritiqueRid, "feedback", Feedback);
        gl.ask("agent1", "refine",
            .concat("Improve:\n", Draft, "\nBased on:\n", Feedback), FinalRid);
        !accept_final(FinalRid).

// Step 4: Accept the refined version
+!accept_final(FinalRid)
    :   gl.valid(FinalRid, true)
    <-  gl.candidate(FinalRid, Cid);
        gl.accept(Cid);
        gl.field(FinalRid, "text", FinalText);
        .print("Final accepted:\n", FinalText).

How iterative refinement improves quality

Each step is a separate sub-goal with its own context guard (gl.valid). The agent chains: Draft → Critique → Refine → Accept. Only the final refined output is accepted. The draft and critique are intermediate — never adopted as beliefs. Each step goes through the full governance pipeline: validation, candidate creation, and traceability.
