How it Works

This page describes the runtime path of one governed resource request.

Request lifecycle

1. Agent program reaches a point where external generative support is needed.
↓
2. Platform adapter converts the host-language call into a Generative Layers request.
↓
3. Governance boundary checks policy, context, limits, admissibility, and logging requirements.
↓
4. Provider invokes the selected resource: LLM, API, tool, service, or assessment module.
↓
5. Framework normalises the result into status, content, metadata, warnings, and audit information.
↓
6. Agent receives the result and decides whether to use, reject, verify, store, or act on it.

Inputs and outputs

Input

A structured request from the agent platform, including task type, payload, context, and policy-relevant metadata.

Output

A structured result, not a direct modification of the agent's belief base, plan library, or intention stack.

Failure handling

A request may be denied, fail, time out, or return warnings. These outcomes are part of the result and should be visible to the agent platform rather than hidden inside the provider call.

Engineering purpose

The runtime model is intentionally small: adapters translate, governance constrains, providers call resources, and agents retain semantic control.
