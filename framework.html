Core concepts

Agent Layer

The host agent platform and its execution semantics.

  • Autonomous Control: Maintains the main BDI (Belief-Desire-Intention) reasoning loop, managing the agent's internal state.
  • Asynchronous Triggers: Inquires external support asynchronously, maintaining internal stability while waiting for generative responses.
  • Context-Guarded Decision: Employs native agent guard conditions to decide whether to adopt, verify, ignore, or reject the candidate material.

Adapter

The bridge between an agent framework and Generative Layers.

  • Datatype Translation: Maps native agent datatypes (like AgentSpeak terms, structures, and lists) to standard Java parameters and vice versa.
  • Syntactic Integration: Realizes the canonical API as native extensions, such as ASTRA Modules, Jason Internal Actions, or CArtAgO Artifacts.
  • Asynchronous Lifecycle Binding: Manages request session tracking, binding asynchronous responses to the initiating agent context.

Governance Boundary

Policy, limits, audit, and admissibility checks.

  • Decoupled Safety Policies: Enforces budget rules, token limits, rate caps, and content boundaries completely outside of agent logic.
  • Audit Logging & Tracing: Automatically generates globally unique trace IDs to log all request contexts, policies checked, and outbound results.
  • Pre-Adoption Verification: Intercepts responses, validating schemas and active rules before exposing candidate material to the agent.

Provider

The abstract connector to an LLM, tool, API, service, or module.

  • Resource Abstraction: Encapsulates model-specific API calls, tool setups, or service connections under a unified invocation contract.
  • Resilience & Networking: Isolates HTTP requests, handles rate limits (429s), and manages retries/backoffs without blocking agent threads.
  • Unified Interface: Simplifies provider switching (e.g. swapping model endpoints or mock providers) with zero changes to agent logic.

Candidate Material

External output that may be adopted, verified, ignored, or rejected.

  • Isolated State Container: Exposes the generated values to the agent without automatically modifying any internal beliefs or plans.
  • Structured Inspections: Provides safe query boundaries (via valid() and field()) so agents can inspect the data before adoption.
  • Deliberated Incorporation: Merges fields into the agent's belief base only when the agent explicitly adopts the candidate (via accept()).

Architecture internals

The 16 canonical commands are powered by a layered internal architecture. These components are not directly exposed to agent code — they operate behind the adapter boundary, governing every invoke() call.

Generative Bodies & Affordances

Pre-registered body instances, affordance types, and candidate type mapping.

When an agent calls invoke(agent, goal, bodyId, affordance, prompt, required), the framework resolves the body ID to a registered GenerativeBody instance and maps the affordance to a semantic CandidateType. The shorthand ask() uses llm.answer with the ANSWER affordance.

Governance Pipeline

The 5-step kernel pipeline executed on every invocation.

Every call to invoke() or ask() passes through the GovernanceKernel. The kernel orchestrates a strict 5-step pipeline — no step can be skipped, and every step produces an auditable trace.

  1. 1 Policy Gate

    The GovernancePolicy evaluates the request. Returns ALLOW, DENY, or ESCALATE. Denied requests never reach the provider.

  2. 2 Conversation Context

    If a conversationId is provided, past prompt/response turns are prepended to the current prompt for multi-turn dialogue.

  3. 3 Provider Call + Retry

    The GenerativeProvider calls the LLM. If the call fails or validation rejects the output, the RetryPolicy can re-prompt with error feedback appended.

  4. 4 Candidate Creation

    Validated output is wrapped in a governed Candidate with status VALIDATED or INVALID. Both prompt and output are stored as content-addressed blobs.

  5. 5 Trace & Turn Recording

    An immutable TraceRecord is created for audit. On success, the conversation context records the new turn for future multi-turn use.

Kernel SPI Ports

10 pluggable interfaces that define all kernel behaviour.

The GovernanceKernel delegates every decision to replaceable SPI ports defined in KernelPorts. Default implementations are provided by KernelDefaults and InMemoryKernelStores.

GenerativeProvider— LLM backend
GovernancePolicy— pre-generation policy gate
ResponseValidator— post-generation schema check
AdmissibilityChecker— pre-acceptance gate
BlobStore— content-addressed storage
CandidateStore— candidate lifecycle
AssessmentStore— peer assessment records
ResultStore— generation result records

Retry Policy

  • Default: No retries — fail on first invalid output or provider error.
  • With retries: Configurable max attempts. Validation errors and provider exceptions are appended to re-prompts as error feedback, enabling self-correcting generation.
  • Conversation context: Multi-turn dialogue tracking per conversation ID. Successful prompt/response turns are recorded and prepended to future prompts.
