Generative Layers

A Java framework for governed generative resource layers around agent-oriented systems.

Purpose

Generative Layers keeps external generation separate from the agent's core execution loop.

LLM, tool, API, and service outputs are returned as candidate material. The host agent platform decides whether to adopt, verify, ignore, or reject them.

Target platforms

ASTRA

AgentSpeak(TR)-style BDI target.

Jason

AgentSpeak-style BDI target.

JaCaMo

Jason, CArtAgO, and Moise integration.

Integration Details & Specifications

Governance architecture, target platforms, and custom integration adapters.

Generative Layers is built to support secure integration with prominent BDI (Belief-Desire-Intention) agent languages and multi-agent programming platforms. The core framework currently offers native, production-tested integrations for ASTRA, Jason, and the integrated JaCaMo platform (which binds Jason BDI agents, CArtAgO environments, and Moise organizations together).

Compare how the same governance flow (configuring, invoking, validating, and adopting candidate material) is realized natively across each target agent environment:

ASTRA Module

Implemented as a native ASTRA Module written in Java. Commands are mapped to inline custom statements.

Jason Internal Actions

Implemented as standard AgentSpeak Internal Actions in Java. Invoked natively by BDI agents using custom namespace extensions.

CArtAgO Artifact (JaCaMo)

Implemented as a CArtAgO Environment Artifact in Java. Commands are operations called once focused on the artifact, receiving updates natively via observable belief signals.

Infinite Composability — Emergent Complexity

BDI (Belief-Desire-Intention) agent-oriented programming is fundamentally about managing autonomous, goal-directed behavior. The integration patterns here are not hardcoded constraints; they represent emergent cognitive architectures. Because you can implement and orchestrate different invocational commands — such as asking for support, asserting active beliefs, checking state transitions, or auditing admissibility — in arbitrary combinations, the possibilities are unlimited. You are empowered to build any system you can conceive natively — entirely at the plan level without changing a single line of framework code.

Architectural Paradigms & Diverse Research Directions
Project 1 — Multi-Agent Software Engineering Swarm (Autonomous Plan-Execution Loops)
Domain: Automated Code Generation & Quality Control.
How it works: A decentralized multi-agent cognitive system that simulates software engineering roles, cooperating through belief-sharing and automated feedback cycles to continuously produce and audit software assets.
Command Combinations:
  • Product Owner Agent: Receives raw specifications and calls ask(requirements_schema) to decompose the task into atomic user stories.
  • Developer Agent: Takes a user story and calls ask(code_schema) to generate candidate Java code.
  • Tester Agent: Generates corresponding unit tests, compiles the program, and uses check(test_results) to assert that the generated code compiles and all tests pass.
  • Security Auditor Agent: Intercepts the code and calls admissible(policy_rules) to check for credentials leakage or SQL vulnerabilities.
  • Governance Plan: If tests fail or the auditor flags a vulnerability, the agent calls reject(candidate_code) and feeds the error trace back into ask for a self-correcting retry loop. If all checks clear, the agent calls accept(candidate_code) and writes the code to the filesystem.
Project 2 — Self-Optimizing RAG-based Clinical Assistant (Safe Epistemic Foundations)
Domain: Healthcare & Safety-Critical Diagnostics.
How it works: A safety-critical expert system that guards the agent's epistemic beliefs, validating external LLM propositions against verified medical repositories before allowing them to be adopted as internally active beliefs.
Command Combinations:
  • Triage Agent: Takes patient symptoms and calls ask(symptom_summary_schema) to generate optimal lookup keys for medical literature.
  • Search Agent: Queries the database and asserts the retrieved clinical papers into the agent's belief base with assert(retrieved_text).
  • Diagnostic Agent: Synthesizes the patient data and research findings to generate a candidate diagnostic report.
  • Admissibility Policy: A strict rule-checking pipeline running admissible(medical_policies) checks the generated diagnosis. It parses the citations in the candidate material, maps them to the actual papers stored in the agent's beliefs, and verifies that the dosage suggestions match established thresholds.
  • Resolution Plan: If any dosage suggestion is outside safe bounds or a citation is unverified, reject is instantly triggered. Otherwise, the agent calls accept and populates the medical dashboard.
Project 3 — E-Commerce Arbitrage & Negotiating Agent (Decentralized Game-Theoretic Plan Harmonization)
Domain: Autonomous Trading & Game Theory.
How it works: An autonomous financial agent that uses utility-driven reasoning to scout markets, negotiate contracts with wholesale providers, and dynamic safety constraints to prevent balance depletion during high-frequency bidding.
Command Combinations:
  • Scouter Agent: Uses ask(price_trends) to predict demand and inventory values, then updates its beliefs using assert(market_price(item, value)).
  • Negotiator Agent: Bids on supplier assets. It uses ask to generate a persuasive counter-offer, which is sent as candidate material.
  • Safety Guard: Before sending any bid, the agent evaluates the candidate counter-offer against an admissibility policy: admissible(max_budget_limit).
  • Commit Plan: If the supplier's price falls below the agent's threshold (verified via check), the agent triggers accept and registers a belief: assert(contract_finalized(supplier, item, price)), executing an automated API payout.
Project 4 — Autonomous Incident Response Grid (Self-Healing Cyber-Physical Systems)
Domain: IoT, Energy Logistics, & Emergency Mitigation.
How it works: A self-healing grid-management system modeled using BDI agents, coordinating dynamically to identify cyber-physical anomalies, draft load-routing configurations, and execute plans to bypass physical line faults safely.
Command Combinations:
  • Monitor Agent: Constantly receives telemetry. If an anomaly is registered, it triggers a goal to resolve the outage.
  • Planner Agent: Uses ask(grid_routing_schema) to generate a candidate rerouting sequence around the damaged substation.
  • Validation Guard: Runs a simulation to verify grid load. It checks if the candidate route is safe by calling admissible(grid_capacity_thresholds).
  • Decision Loop: If overloading is predicted, the plan triggers reject and prompts ask to generate a load-shedding route instead. Once a route is validated, accept commits the changes, triggering physical smart switches on the grid.
Project 5 — Interactive NPC Dynamic Dialogue System (Emergent Narrative & Social Interaction)
Domain: Gaming & Creative Entertainment.
How it works: Virtual social agents equipped with distinct personality variables, social relationship beliefs, and dynamic dialogic generation plans, producing contextually coherent narrative interactions.
Command Combinations:
  • Perception Agent: Records player choices and stores them as memory beliefs: assert(player_relation(NPC, player, hostile)).
  • Dialogue Agent: When the player approaches, the NPC calls ask to generate a dialogue draft based on its current mood, personality traits, and past memories.
  • Lore Auditor: The game director agent calls admissible(lore_guidelines) to check that the draft dialogue does not conflict with established game lore or contain inappropriate speech.
  • Adopt & Speak: If approved, the agent calls accept and uses assert to record the dialogue line into the narrative history before playing the voice line.
