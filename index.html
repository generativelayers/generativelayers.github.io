Generative Layers

A Java framework for governed generative resource layers around agent-oriented systems.

Purpose

Traditional agent programming relies on deterministic, goal-directed symbolic state—a model that clashes directly with the unpredictable, probabilistic nature of LLMs.

When agents directly commit raw LLM outputs to memory, they suffer state pollution, breaking execution invariants and safety guarantees. Generative Layers solves this by quarantining LLM interactions inside isolated, multi-phase transactions. Unstructured outputs are sandboxed as uncommitted candidate material (cand_) and are only adopted once they satisfy the agent's symbolic logic and admissibility constraints. This ensures that hallucinations and formatting anomalies are rejected at the gate, keeping the host agent's reasoning loop deterministic and secure.

Integration Details & Specifications

Governance architecture, target platforms, and custom integration adapters.

Generative Layers is built to support governed integration with prominent BDI (Belief-Desire-Intention) agent languages and multi-agent programming platforms. The core framework currently offers implemented cross-platform integrations for ASTRA, Jason, and the integrated JaCaMo platform (which binds Jason BDI agents, CArtAgO environments, and Moise organizations together).

Compare how the same governance flow (configuring, invoking, validating, and adopting candidate material) is realized natively across each target agent environment:

ASTRA Module

Implemented as a native ASTRA Module written in Java. Commands are mapped to inline custom statements.

Jason Internal Actions

Implemented as standard AgentSpeak Internal Actions in Java. Invoked natively by BDI agents using custom namespace extensions.

CArtAgO Artifact (JaCaMo)

Implemented as a CArtAgO Environment Artifact in Java. Commands are operations called once focused on the artifact, receiving updates natively via observable belief signals.

Composable Governance Patterns

Rather than hardcoding security policies, Generative Layers introduces Neuro-Symbolic Transactions. By structuring LLM queries into a formal lifecycle—binding providers, invoking resources, sandboxing candidates, registering judgements, and verifying admissibility—the framework allows developers to orchestrate custom cognitive boundaries. This enables BDI agents to arbitrate neural suggestions using pure symbolic logic before committing them to active state.

Architectural Paradigms & Diverse Research Directions
Project 1 — Multi-Agent Software Engineering Swarm (Governed Code Generation & Self-Correcting Execution Loops)
Domain: Governed Software Synthesis & Quality Verification.
How it works: A decentralized BDI swarm simulating developer roles, cooperating via belief sharing and automated compiler/linter feedback to continuously synthesize and verify software artifacts under transactional containment.
Neuro-Symbolic Lifecycle Flow:
  • Product Owner Agent: Ingests specifications and calls call() to decompose requirements into structured backlog items.
  • Developer Agent: Receives a backlog item and invokes a neural provider to generate candidate code, immediately quarantining the file contents inside a candidate() transaction.
  • Tester Agent: Orchestrates execution-level testing, compiling the candidate and registering a judge() based on unit test success.
  • Security Auditor Agent: Audits the quarantined candidate code for common vulnerability signatures, registering an independent safety judge().
  • Governance Decoupling: The host agent calls decide() to aggregate all peer judgments. If inadmissible, reject() is called, and the error trace triggers a self-correcting plan. If admissible, accept() commits the candidate code to the codebase.
Project 2 — Self-Optimizing RAG-based Clinical Assistant (Epistemic Governance & Safety-Critical Verification)
Domain: Clinical Decision Support & Epistemic Safety.
How it works: A diagnostic assistant that guards the agent's epistemic belief base, isolating and verifying generated medical recommendations against trusted scientific repositories before committing them as active beliefs.
Neuro-Symbolic Lifecycle Flow:
  • Triage Agent: Evaluates symptom profiles and invokes a neural provider to generate targeted query keys for medical databases.
  • Diagnostic Agent: Dispatches patient telemetry through call() to draft a diagnostic report, immediately quarantining the recommendation as a raw candidate().
  • Safety Auditor Agent: Inspects specific sub-fields of the candidate using get(), extracting dosage recommendations and literature citations for out-of-band verification.
  • Epistemic Validation: Cross-references dosage bounds and citations against localized clinical knowledge, registering a safety judge().
  • Symbolic Adoption: If decide() returns true, the agent runs accept() to merge the diagnosis into its active belief base. Otherwise, it executes reject() and logs the safety violation.
Project 3 — E-Commerce Arbitrage & Negotiating Agent (Utility-Driven Bidding under Transaction Boundaries)
Domain: Autonomous Financial Negotiation & Budget Enforcement.
How it works: An automated trading agent that negotiates wholesale supplier contracts, enforcing hard financial spending limits to prevent capital depletion during high-frequency bidding.
Neuro-Symbolic Lifecycle Flow:
  • Scouter Agent: Analyzes demand indicators and uses call() to extract market price trends, validating the output before updating inventory estimates.
  • Negotiator Agent: Conducts bargaining rounds, prompting a model to draft a counter-offer that is immediately quarantined as a candidate().
  • Budget Monitor Agent: Parses the proposed bidding amount with get() and runs a check against the agent's current liquid capital.
  • Governance Verification: Registers a budget-compliance judge(). If the bid violates financial limits, it triggers reject(), reverting the agent to a more conservative negotiation plan.
  • Contract Execution: Once decide() confirms admissibility, the agent calls accept(), committing the offer to memory and initiating the external transaction.
Project 4 — Autonomous Incident Response Grid (Smart-Grid Load Routing & Simulator-in-the-Loop Validation)
Domain: Cyber-Physical Resilience & Real-Time Incident Response.
How it works: A smart-grid logistics platform where BDI agents detect electrical faults, generate alternative load-routing configurations, and validate them in simulations before enacting physical grid changes.
Neuro-Symbolic Lifecycle Flow:
  • Telemetry Monitor Agent: Detects physical grid anomalies and triggers an emergency recovery plan to bypass the damaged sector.
  • Logistics Planner Agent: Dispatches the fault telemetry through call() to generate a candidate rerouting sequence, isolating the raw plan as a candidate().
  • Grid Simulator Agent: Simulates the electrical load distribution of the proposed route, evaluating whether lines will exceed safety limits.
  • Admissibility Auditing: Submits the simulator outcomes as a judge(). If decide() rejects the path, the agent calls reject() and falls back to a load-shedding plan.
  • Physical Commitment: Upon successful validation, the agent runs accept() and translates the committed configuration into physical smart-switch command signals.
Project 5 — Interactive NPC Dialogue Engine (Emergent Narrative & Lore Consistency Auditing)
Domain: Emergent Narrative & Interactive Agents.
How it works: Virtual game characters with distinct personality profiles and social beliefs, generating contextually appropriate dialogue while auditing responses to ensure compliance with established world lore and behavioral bounds.
Neuro-Symbolic Lifecycle Flow:
  • Perception Agent: Observes player behavior and updates social relationship beliefs (e.g., trust metrics, historical interactions) within the BDI belief base.
  • Dialogue Agent: Generates a character-appropriate response draft via call(), immediately isolating the unstructured speech string in a candidate() container.
  • Lore Monitor Agent: Checks the candidate's speech text against the world database to prevent anachronisms or breaks in narrative consistency.
  • Content Safety Checker: Validates that the response conforms to toxicity and family-friendly thresholds, submitting its verdict as a judge().
  • Speech Committal: The agent calls decide(). If admissible, accept() transitions the text to active dialogue; otherwise, reject() discards the draft, prompting a silent conversational retry.
